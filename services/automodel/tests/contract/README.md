## Directory Layout

Submit-time job JSON (including W&B / MLflow integrations) lives in
`plugins/nemo-automodel/tests/fixtures/`.

Input configs are grouped by model so the generation script downloads each model only once:

```
input_configs/
    llama-3.1-8b/              # one subdirectory per model
        llama_3_1_8b_full_sft_tp.json
    llama-3.2-1b/
        llama_3_2_1b_full_sft.json
        llama_3_2_1b_lora.json
        ...
    nemotron-nano/
        nemotron_nano_full_sft.json
        ...
    embed-1b/
        ...
    gpt-oss/
        ...
output_configs/                # flat — one YAML per input JSON
    llama_3_2_1b_lora.yaml
    ...
```

## Key Notes

- **Model paths**: Input configs use HuggingFace model IDs (e.g. `meta-llama/Llama-3.2-1B-Instruct`). The generation script downloads model configs and tokenizers from HF once per model group.
- **Dataset paths**: Configs reference sample datasets in `./sample-datasets`. These are small (100 train / 10 val) subsets for quick validation.
- **Sequence packing**: Packing configs have `packed_sequence.packed_sequence_size` computed from actual dataset token length statistics.
- **Nemotron Nano**: Uses `trust_remote_code: true` since the model has custom architecture code (`NemotronHForCausalLM`).
- **Embedding model**: Uses Automodel's biencoder recipe (`TrainBiencoderRecipe`) with retrieval dataset format, not the LLM recipe.
- **Generation environment**: Run `generate_configs.py` in the **nmp-automodel-training** image on a GPU pod. Compiling configs requires `nemo_automodel` and its dependencies which are only available in that image.

## Running the Configs

All configs are in `output_configs/` and can be passed directly to the Automodel finetune scripts.

### LLM configs (all models except embedding)

Single GPU:
```bash
python examples/llm_finetune/finetune.py --config /path/to/output_configs/<config>.yaml
```

Multi-GPU (e.g. nemotron_nano_full_sft and gpt_oss_full_sft need 8 GPUs):
```bash
torchrun --nproc_per_node=<NUM_GPUS> examples/llm_finetune/finetune.py --config /path/to/output_configs/<config>.yaml
```

### Embedding configs

```bash
python examples/biencoder/finetune.py --config /path/to/output_configs/<config>.yaml
```

### Quick reference — all configs

## Regenerating Configs

When Customizer's config compilation logic changes, regenerate all configs:

```bash
python3 generate_configs.py --all
```

Or regenerate a single config:

```bash
python3 generate_configs.py input_configs/nemotron-nano/nemotron_nano_lora.json -o output_configs/nemotron_nano_lora.yaml
```

### Prerequisites

The script needs `nemo_automodel` and its dependencies. Run it inside the **nmp-automodel-training** container image, which has everything pre-installed:

```bash
# On a GPU pod with the nmp-automodel-training image
cd services/automodel/tests/contract
python generate_configs.py --all
```

The script also needs HuggingFace access for model configs/tokenizers. Set `HF_TOKEN` for gated models.

## CI Automation

These fixtures are retained for the Platform-Deploy contract-test pipeline. Platform no longer wires this validation as a local GitLab CI job, but the check still regenerates all configs and compares them against the committed versions. If any config is stale, the Deploy-owned job fails with a summary of which configs changed.

### What to do when the validation fails

The job fails when the committed output configs don't match what the current code generates. This means the config compilation code changed but the output configs weren't regenerated. To fix it:

1. **Get a GPU pod** with the `nmp-automodel-training` image (the script needs `nemo_automodel` which is only available in that image).

2. **Regenerate all configs:**
   ```bash
   cd services/automodel/tests/contract
   python generate_configs.py --all
   ```

3. **Review the diff** to understand what changed:
   ```bash
   git diff output_configs/
   ```

4. **Commit and push** the updated output configs:
   ```bash
   git add output_configs/
   git commit -m "regenerate automodel contract configs"
   git push
   ```

5. **Notify the Automodel team** if the changes are significant (new fields, renamed targets, removed sections). They should run the updated configs against their `finetune.py` to catch any incompatibilities.

### Checking configs locally

```bash
# Verify all output configs match current code (exits non-zero if stale)
python3 generate_configs.py --check
```
